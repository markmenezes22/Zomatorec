# AI-Powered Restaurant Recommendation System Implementation Plan

This document outlines the phase-wise implementation plan for the Zomato-inspired AI restaurant recommendation service, based on the `architecture.md` and `context.md` requirements.

## Proposed Changes

### Phase 1: Data Ingestion Layer
Set up the core data pipeline to load and cache the Hugging Face dataset.

- **`src/config.py`**: Define environment variables, constants, and budget mapping thresholds.
- **`src/models/restaurant.py`**: Define the canonical `Restaurant` dataclass/schema.
- **`src/data/loader.py`**: Implement dataset fetching from Hugging Face (`datasets` library).
- **`src/data/preprocessor.py`**: Implement data cleaning, string normalization, cuisine parsing, and numeric coercion.
- **`src/data/repository.py`**: Implement the in-memory cache and query interface for the dataset.

---

### Phase 2: User Input & Deterministic Filtering
Build the logic to validate user preferences and heuristically narrow down candidates before they hit the LLM.

- **`src/models/preferences.py`**: Define the `UserPreferences` input schema and validation logic.
- **`src/services/filter.py`**: Implement the deterministic filter pipeline (Location → Budget → Min Rating → Cuisine) and top-K candidate selection.

---

### Phase 3: Recommendation Engine (LLM Layer)
Integrate the Groq API to rank candidates and generate personalized natural language explanations.

- **`src/models/recommendation.py`**: Define `Recommendation` and `RecommendationResponse` response models.
- **`src/services/prompt_builder.py`**: Implement the logic to assemble context, candidates, and user preferences into a robust JSON-enforced prompt for the LLM.
- **`src/services/llm_client.py`**: Implement the client adapter for the `groq` SDK with error handling.
- **`src/services/recommendation.py`**: Orchestrate the filtering, prompt building, LLM invocation, response parsing, and enrichment with structured data.

---

### Phase 4: Backend API Layer
Expose the recommendation engine via a RESTful API for decoupling backend logic from the frontend UI.

- **`src/api/routes.py`**: Define FastAPI endpoints (e.g., `POST /api/recommendations`) to accept `UserPreferences` and return structured recommendations.
- **`src/main.py`**: Set up a FastAPI application instance, configure CORS middleware, and register routes.

---

### Phase 5: High-Quality Frontend UI
Build a premium, dynamic web application to deliver an excellent user experience, utilizing modern frontend architectures instead of basic prototyping tools.

- **Frontend Tech Stack**: Vite + React + Tailwind CSS (v3). (Using Tailwind to directly integrate the design generated by Google Stitch in `stitch_gourmet_ai_concierge`).
- **Design System**: Implement the Hyper-Modern Glassmorphism aesthetics defined in `DESIGN.md`, utilizing the custom Tailwind config provided in the Stitch output.
- **`frontend/`**: Directory for the standalone frontend workspace, decoupled from the backend.
- **Components**: Build robust React components:
  - **Sidebar Form**: Intuitive preference collection including a dropdown for Location, budget toggle, cuisine multi-select, and rating slider.
  - **Main Display**: Dynamic recommendation displays using the provided animated result cards and summary banner.
  - **States**: Implement loading and empty states as designed.

---

### Phase 6: Hardening & Verification
Final polish, test coverage, and documentation.

- **`tests/test_filter.py`**
- **`tests/test_preprocessor.py`**
- **`tests/test_recommendation.py`**
  Implement unit tests covering deterministic filters, data parsing, and valid/invalid LLM response handling.
- **`README.md`**: Document system requirements, environment setup (`GROQ_API_KEY`), run instructions, and architecture.

## Verification Plan

### Automated Tests
- Run `pytest tests/` to verify all pre-filtering heuristics, schema mapping, and fallback logic without requiring live LLM calls.

### Manual Verification
- Boot the FastAPI backend and Vite frontend locally.
- Execute multiple user journeys with varying preferences (e.g., highly specific constraints vs. broad constraints).
- Verify the LLM rationale matches the user preferences and the data characteristics of the suggested restaurants.
- Validate that the UI design is premium, responsive, features micro-animations, and correctly handles empty result states or invalid inputs.
